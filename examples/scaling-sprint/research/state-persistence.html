State Persistence: Remote Farmer Reliability

Research | 2026-03-12 | 5 claims | r038-r042

The Remote Farmer server stores all state in memory. A crash or restart means total loss: trust levels, session rules, activity audit trail, and agent tracking all vanish. This research maps what should be persisted, evaluates approaches within the zero-dependency constraint, and recommends a solution.

The Risk: Total State Loss

HIGH All State In Memory

The sessions Map (line 76 of server.mjs) holds every piece of server state. On process death:

  • Trust levels reset to paranoid (user must reconfigure)
  • Session rules wiped (auto-approve patterns lost)
  • Activity log gone (no audit trail)
  • Pending permissions silently dropped (no deny sent to Claude Code)
  • Agent tracking lost

r038

MEDIUM No Signal Handlers

The server has no SIGTERM or SIGINT handlers. Even a graceful Ctrl+C or kill provides no opportunity to save state or deny pending requests cleanly. r041

Connection to Security (r034)

The security research noted that token rotation requires a server restart, which "loses in-memory session state." State persistence directly reduces the cost of restarts, making token rotation and other security measures more practical.

State Classification

Not all state deserves persistence. The key insight: pending permissions are ephemeral by nature (they hold HTTP response objects), while trust levels and rules are user configuration that should survive restarts. r040

Approach: JSON Snapshots

The recommended solution uses only Node built-in modules (fs, process), satisfying the zero-deps constraint d004. r039

How It Works

  • Write recoverable state to farmer-state.json using fs.writeFileSync
  • Atomic write via write-to-temp-then-rename pattern prevents corruption on crash mid-write
  • Trigger: on trust/rule changes + periodic 30s flush + graceful shutdown
  • On startup: load farmer-state.json to restore trust levels, rules, and activity
// Atomic write: temp file then rename
const tmp = STATE_PATH + '.tmp';
writeFileSync(tmp, JSON.stringify(state, null, 2));
renameSync(tmp, STATE_PATH);

// On startup
if (existsSync(STATE_PATH)) {
  const saved = JSON.parse(readFileSync(STATE_PATH, 'utf8'));
  for (const s of saved.sessions) {
    const session = getSession(s.id, s.cwd);
    session.trustLevel = s.trustLevel;
    session.sessionRules = s.sessionRules || [];
    session.activity = s.activity || [];
  }
}

Performance

Full state is <100KB (200 activity entries + rules + trust across 10 sessions). Write takes <1ms on SSD. No measurable impact on request handling.

Graceful Shutdown

Essential companion to persistence: handle SIGTERM and SIGINT to save state and cleanly deny pending requests. r041

function gracefulShutdown(signal) {
  // 1. Deny all pending permissions
  for (const [sid, session] of sessions) {
    for (const [reqId, entry] of session.pending) {
      entry.resolve({
        allow: false,
        reason: `Server shutting down (${signal})`
      });
    }
  }
  // 2. Flush state to disk
  saveState();
  // 3. Notify dashboard clients
  broadcast({ type: 'server_shutdown', data: { signal } });
  // 4. Close server, force exit after 3s
  server.close(() => process.exit(0));
  setTimeout(() => process.exit(1), 3000);
}

process.on('SIGTERM', () => gracefulShutdown('SIGTERM'));
process.on('SIGINT', () => gracefulShutdown('SIGINT'));

Why Not Event Sourcing?

Append-Only Log: Over-Engineered Here

An append-only event log (writing each state change as a JSON line) provides zero data loss. But the complexity cost is high: r042

  • Requires log compaction to prevent unbounded growth
  • Startup replays the entire log to reconstruct state
  • Schema evolution is harder (every event format is permanent)
  • All this for <100KB of state that writes <1/sec

Verdict: JSON snapshots are sufficient. Consider event sourcing only if state exceeds 1MB or write frequency exceeds 10/sec. Neither is expected for a permission dashboard.

SQLite: Disqualified

better-sqlite3 would provide ACID transactions and query capability, but it is an npm dependency. Violates d004 (zero npm deps constraint). Not considered further.

Implementation Plan

Phase 1: JSON Snapshots + Signal Handlers (~40 LOC)

  • Add saveState() and loadState() functions
  • Call saveState() in handleTrustLevel, handleRules, and on 30s interval
  • Call loadState() after loadClaims() on startup
  • Add SIGTERM/SIGINT handlers with gracefulShutdown()

Phase 2: Dashboard Indicator (optional)

  • Show "last saved: Xs ago" in dashboard footer
  • Add --save-interval CLI flag for configurable flush frequency
  • Broadcast state_saved SSE event for dashboard to display

Data Loss Window

Worst case: 30 seconds of activity data lost on SIGKILL (hard crash). Trust levels and rules are saved immediately on change, so they are never lost except on SIGKILL during the write itself (mitigated by atomic rename). For a local development tool, this is acceptable.
