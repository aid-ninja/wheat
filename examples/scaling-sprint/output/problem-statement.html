Wheat Sprint Define Phase 5 constraints

Scaling Farmer + Repo Cartography

How should we scale Remote Farmer for multi-session and multi-sprint support, and what repo structure keeps AI interactions fast as the codebase grows?

The Problem

Remote Farmer works for a single Claude Code session with a single sprint. But real usage needs: multiple sessions in one dashboard, multiple sprints viewable from the phone, and a repo structure that doesn't slow down as it grows.

As sprints accumulate (claims, prototypes, research, output), AI models spend more time searching. We need a cartography system — a self-describing repo that tells Claude Code where things are without scanning everything.

Constraints

Multi-session

Multiple concurrent Claude Code sessions in one dashboard, each identifiable and independently approvable. [d001]

Multi-sprint

Switch between sprints in the remote UI — claims, status, and artifacts for each. [d002]

Cartography

Self-describing repo — new sessions understand structure without full-tree scans. Manifest/index system. [d003]

Performance

Measurably faster AI search with cartography. Must scale to 10+ sprints, 500+ claims. [d005]

Done Looks Like

Multi-session Farmer prototype + repo manifest/index system + measured improvement in AI search time.

Audience

Solo developer scaling to team use.
