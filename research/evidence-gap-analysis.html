Key Finding

3 of 4 are false alarms

audience, scope, quality
Constraints can't be "upgraded"
When a stakeholder says "audience is developers" — that's a decision, not a hypothesis. The evidence tier stated is correct by definition. You can't "test" or "research" a constraint into a higher tier.

No action needed — by design
output-format
This one is a real gap
The risk claim [r007] about server-vs-static tension is at web tier. We've already built both approaches — the static HTML works at file://. We just haven't recorded a prototype finding about it.

Upgradeable via /prototype test
Compiler gap
The compiler should know the difference
Topics dominated by constraints should use a different metric — "constraint completeness" rather than evidence tier. A topic with 5 well-defined constraints is not "weak."

Compiler improvement proposed
[r009] [r010] [r011]
Recommendations

Three paths forward

1. Test output-format

Open output/dashboard.html via file://. Verify zero network requests, all CSS/JS inline, works offline. Record as prototype claim with tested evidence. This clears the one real warning.

2. Improve the compiler

Add constraint-aware coverage analysis. Topics where >50% of claims are type constraint or feedback should not emit W_WEAK_EVIDENCE. Instead, emit W_CONSTRAINT_ONLY as an informational note. [r012]

3. Accept and move on

The simplest path: these warnings are non-blocking. The compiler status is READY. The warnings exist to surface gaps — once you've analyzed them and confirmed they're not real gaps, the warnings have done their job.

Impact of path 1
Warnings drop from 4 to 3
output-format upgrades to tested. The 3 constraint-only topics remain flagged (acceptable noise until compiler is improved).
Impact of path 2
Warnings drop from 4 to 0
Compiler becomes aware of the constraint/evidence distinction. All 4 topics stop generating false positives. Better signal-to-noise for future sprints.
Summary

Most "weak" topics
aren't actually weak

Constraints and feedback are inherently stakeholder-level claims. The compiler's evidence tier system was designed for testable hypotheses, not for definitional boundaries.

One real gap: output-format can be upgraded by testing the static HTML artifact.
Three false alarms: audience, scope, quality are correctly at stated-level.
One improvement: compiler should distinguish constraints from testable claims.
