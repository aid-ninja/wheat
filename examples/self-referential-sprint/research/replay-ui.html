Wheat Research

Replay UI

An interactive timeline that plays back a Wheat sprint commit-by-commit — claims appearing, evidence upgrading, coverage growing — in a single self-contained HTML file.

Compiled by Wheat · r042–r047
The Problem

Sprints tell a story.
Git log buries it.

A Wheat sprint is a sequence of research decisions — each commit changes the claim landscape. But git log --oneline shows flat text, not the shape of the investigation.

We want to watch the sprint unfold: see the first claims appear, evidence accumulate, conflicts emerge and resolve, coverage fill in topic by topic.

Use Cases
Learning
New developer watches a sprint replay to understand how Wheat works by seeing it work.
Use Cases
Archival
Reset main to start fresh, but keep the old sprint watchable as a self-contained artifact.
Core Pattern

Pre-Computed Frames

The dominant pattern across Observable, D3, and data journalism: bake an array of state snapshots, render one at a time, animate between them.

1. Frame Array
frames[i] = full state
Each git commit that touches claims.json becomes one frame. The frame contains the compiled claims, coverage map, stats, and delta from the previous frame. Embedded as inline JSON. [r042]
2. Stateless Render
render(frame) → DOM
One function that takes a frame object and updates the entire UI to match. No accumulated state, no side effects. The frame is the single source of truth.
3. Controller
play / pause / scrub
Manages currentFrame, playing, speed. Drives a requestAnimationFrame loop. The scrubber slider and the play button both write to the same state.

[r042] For ~30 frames × ~100 claims, inline data is well under 1MB. The HTML file is fully self-contained.

Data Pipeline

From Git to Frames

Reuses the existing compiler infrastructure — no new diff engine needed. [r046]

git log --oneline -- claims.json          # list commits

git show <hash>:claims.json               # snapshot at each commit

node wheat-compiler.js --input snapshot   # compile each snapshot

node wheat-compiler.js --diff prev curr   # delta between consecutive

→ const FRAMES = [ ... ]               # bake into HTML

Frame Object

{

  compilation: {
    claims, coverage, stats

      field: "evidence",
      from: "web", to: "tested" }]
  },
  milestone: "entered prototype phase"
}
Key Insight
Compiler is the diff engine
The compiler's diffCompilations() already computes claim adds, removes, status changes, coverage changes, and conflict graph changes. We just need to call it in a loop and collect the results.
Interaction

Timeline Scrubber

Three components: range slider, play/pause, keyboard shortcuts. [r043]

Range Input
<input type="range" step="1"> for discrete frame snapping. Custom-styled thumb and track. JS-driven gradient fill for progress.
Play / Pause
requestAnimationFrame loop. Auto-pauses when user grabs slider. Speed: 1x, 2x, 4x, instant.
Keyboard
←/→ step ±1 frame. Space play/pause. Home/End jump to first/last.
Transitions

Animating Claim Changes

CSS transitions handle most cases. FLIP technique for reordering. [r044]

Change Technique Duration
Claim appears opacity: 0→1 + translateY(10px→0) 300ms, staggered
Claim removed opacity: 1→0 + scale(0.95) 200ms
Evidence upgrade Background color pulse (amber flash) 400ms
Reorder FLIP: measure → DOM change → inverse transform → animate 350ms
FLIP Technique
First: measure element positions. Last: apply DOM change, measure again. Invert: apply inverse transform (element appears in old position). Play: remove transform with CSS transition → element animates to new position. Requires requestAnimationFrame between Invert and Play.
Counters
Animated numbers via requestAnimationFrame with exponential ease-out: current += (target - current) * 0.15. Converges visually in ~15 frames (~250ms at 60fps).
Diffs

Showing What Changed

Simple set operations on claim IDs — no generic JSON diff library needed. [r045]

Algorithm

prevIDs = new Set(prev.claims.map(c => c.id))
currIDs = new Set(curr.claims.map(c => c.id))

added   = [...currIDs].filter(id => !prevIDs.has(id))
removed = [...prevIDs].filter(id => !currIDs.has(id))

upgraded = [...currIDs]
  .filter(id => prevIDs.has(id))
  .filter(id => {
    prev_c = prev.claims.find(c => c.id === id)
    curr_c = curr.claims.find(c => c.id === id)
    return prev_c.evidence !== curr_c.evidence
        || prev_c.status  !== curr_c.status
  })

Display Layers

Summary Bar
Always visible: "Frame 12: +3 claims, 1 upgraded, new topic: dogfooding"
Inline Badges
Each claim gets a colored dot when it was added or changed in the current frame. Fades after 2 frames.
Milestone Detection
Auto-detect: phase transitions, first "ready" compilation, evidence tier jumps, conflict resolution events. Shown as highlighted markers on the timeline.

Risks

What Could Go Wrong

Git Dependency
Frame extraction needs git history
git show <hash>:claims.json only works in a repo with intact history. Force-push or rebase destroys frames. Mitigation: the generated HTML is self-contained and survives repo deletion. [r047]
File Size
Inline data scales with frames × claims
30 frames × 100 claims ≈ 300KB of JSON. Manageable. But a 500-commit sprint with 1000 claims could hit several MB. Compress by storing only deltas instead of full snapshots.
Browser Compatibility
View Transitions API is new
Baseline Oct 2025, but older browsers won't support it. CSS transitions are the safe fallback. FLIP technique works everywhere. Plan: use View Transitions with graceful degradation.
Compiler Version Skew
Recompiling old claims with today's compiler
The current /replay design applies the latest compiler to historical data. This is intentional (shows "what would have happened"), but means the replay doesn't perfectly reconstruct the original compilation output.
