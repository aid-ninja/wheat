The Core Primitive
Think of it like a lab notebook
Everything is a Claim
Every finding, every constraint, every risk — wheat captures it as a typed, evidence-graded claim. Not a paragraph in a doc. Not a message in a thread. A structured record with an ID, a type, a source, and an evidence tier.

Claims are the atoms of your research. They can be challenged, corroborated, superseded, or resolved — but they never disappear. The trail is always there.
Key insight: Structured claims make research auditable. Anyone can trace a decision back to the evidence that produced it.

Why Quality Matters
Think of it like a courtroom
Not All Evidence is Equal
A claim backed by "someone said so" is not the same as one backed by a passing test suite. Wheat enforces this with five evidence tiers, from hearsay to production metrics.

When two claims conflict, the compiler doesn't flip a coin. Higher-tier evidence wins. A documented finding outranks a web search. A tested prototype outranks documentation.
Key insight: The evidence gap is structural, not subjective. Claims with citations and test results naturally outrank claims backed only by reasoning.

The Engine
Think of it like a quality inspector
7 Compiler Passes
Every time you run a command, wheat compiles your claims through seven deterministic passes. Schema validation. Type checking. Evidence sorting. Conflict detection. Auto-resolution. Coverage analysis. Readiness check.

The compiler is the enforcement layer. If your research has unresolved conflicts, missing fields, or coverage gaps — it blocks artifact generation until you fix them.
Key insight: Same claims in = same compilation out. The certificate hash proves it. Your research is reproducible by design.
1Schema ValidationRequired fields, valid IDs
2Type CheckingValid claim types, statuses, tiers
3Evidence SortingDeterministic ordering by tier
4Conflict DetectionSame topic, contradicting claims
5Auto-ResolutionHigher evidence wins, ties flagged
6Coverage AnalysisDiversity, corroboration, gaps
7Readiness CheckBlock or pass for artifact gen
When Evidence Disagrees
Think of it like two witnesses in court
Conflict Resolution
Two claims about the same topic. One says "SSE scales to 10k connections." The other says "SSE caps at 1k." The compiler catches this automatically.

If one claim has stronger evidence — tested vs. stated — the compiler resolves it. If they're tied, it flags the conflict and blocks until you manually adjudicate with /resolve. No silent contradictions ship.
Key insight: Most research tools let contradictions live in different docs forever. Wheat surfaces them as compiler errors you must resolve.
r008 web SSE connections cap at ~1,000 per browser due to HTTP/1.1 connection limits.
conflict detected
p003 tested Prototype sustained 8,200 concurrent SSE connections on a single Node.js process.
↓
p003 wins — tested evidence (tier 4) outranks web evidence (tier 2)
The Audit Trail
Think of it like a time capsule at every step
Git is the Event Log
Every slash command that modifies claims auto-commits to git. The commit message is the event: what command ran, what claims were added or changed.

git log --oneline claims.json = your sprint history. You can replay the entire research journey, see exactly when a finding entered, when it was challenged, when it was resolved.
Key insight: No separate audit system needed. Git is the ledger. The /replay command turns it into an interactive time-travel viewer.

What Comes Out
Think of it like a compiler output
Claims Compile to Artifacts
Claims go in. Decision briefs, presentations, dashboards, and handoff packages come out. Every statement in every artifact traces back to a claim ID. Nothing is uncited.

Artifacts are self-contained HTML files — no external dependencies, no build step. Open them in any browser. Share them with stakeholders. The compilation certificate at the bottom proves integrity.
